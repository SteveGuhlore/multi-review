multi-review

cheat sheet · dashboard

Three AI models (Claude · Codex · Gemini) review your code together, so one model never gets the last word. Language-agnostic, zero-config, safe by default. Rule of thumb: have code → /multi-review; want code → /goal; add --apply only when it should actually edit.

/multi-review

the auditor — check what exists

All available models read your code, debate, and agree on the real issues. Report by default; opt in to auto-fix the safe ones.

/multi-review full /multi-review path server --apply
for: audits · PR review · security pass

/goal

the builder — make / change code

Plans it, builds it, gates it, then runs the auditor for you — looping until it converges or hits a cap.

/goal "add CSV export" --checkpoints /goal "fix the date bug" --auto --apply
for: features · bug fixes · refactors

Which do I use?
  • Audit my PR before merge → /multi-review
  • Audit + clean up safe nits → /multi-review full --apply
  • Build a feature, stay in control → /goal "…" --checkpoints
  • Small fix, hands-free → /goal "…" --auto --apply
  • Plan before building → "help me plan X" (helpmecode)
  • Prove it's safe — no AI/tokens → /goal --gates-only
🛡 Safety rails
✓Auto-fix runs only on a branch — refuses on main/master
✓Report-only unless you add --apply
✓Critical / security findings are never auto-edited → PLAN.md
✓Each fix is a revertible commit; tests go red → it reverts

Always read PLAN.md — the heightened-review queue for anything the loop refused to auto-fix.
